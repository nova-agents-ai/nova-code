## 3. retry / rate-limit / abort 子集

### 3.1 `src/services/api/withRetry.ts`

签名：

```ts
export interface WithRetryOptions {
  readonly maxAttempts?: number;      // 默认 3
  readonly initialDelayMs?: number;   // 默认 500
  readonly maxDelayMs?: number;       // 默认 16_000
  readonly signal?: AbortSignal;
  /** DI：覆盖默认的 setTimeout，便于单测。 */
  readonly sleep?: (ms: number) => Promise<void>;
}

export async function withRetry<T>(
  fn: (attempt: number) => Promise<T>,
  options?: WithRetryOptions,
): Promise<T>;
```

**实现范围**：

- HTTP status 级重试：`{429, 502, 503, 504, 529}`
- 网络错误级重试：`{ECONNRESET, ECONNREFUSED, ETIMEDOUT, ENETUNREACH, EAI_AGAIN, EPIPE}`
- 延迟策略：`Retry-After` 头优先 → 否则指数退避 `initialDelayMs × 2^(attempt-1)` capped by `maxDelayMs`，±25% jitter
- `AbortError` 透传、不重试；睡眠期间周期性检查 `signal.aborted`

### 3.2 `src/services/api/errorUtils.ts`

- `isAbortLikeError(error)`：识别 `AbortError` / `AbortSignal` 触发链
- `isRetryableError(error)`：status 或 network code 命中 retry 白名单
- `getRetryAfterMs(error)`：从 APIError 的 `headers` 解析 `Retry-After`（支持 Web Fetch `Headers` 对象与 `Record<string, string>` 两种形态）

### 3.3 单测（11 个用例全过）

覆盖 Retry-After header / status 429 / status 529 / 网络错误 ECONNRESET / abort 中断等值得单挑的场景。

## 5. debug sink 按 session 切分（M0 债 #4 的预埋）

### 现状登记

M1.5 时点 `createFileDebugSink` 按 `ask-<timestamp>-<pid>.log` 命名，一次 ask = 一个进程 = 一个日志文件，等同单 session 切分。

### M1.5 实际动作

- `CreateFileDebugSinkOptions` 新增 `sessionId?: string`
- `buildDebugLogFileName(now, pid, sessionId?)` 签名扩展
  - 传 sessionId：`ask-<ts>-<sessionId>.log`
  - 不传：保持 `ask-<ts>-<pid>.log`（向后兼容）
- M2 chat REPL 引入多会话时，调用方直接传 sessionId 即可自动切分，无需再改 sink

### 单测新增

- 传 sessionId → 文件名包含 sessionId
- 不传 sessionId → 文件名包含 pid（现有行为）
- 传 undefined 显式等价于不传
